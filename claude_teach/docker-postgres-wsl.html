Docker + PostgreSQL no WSL

Guia completo para o futuro eu — do zero até o banco rodando.

Contexto

Este guia documenta o processo de instalar o Docker corretamente no WSL2 (Ubuntu) e subir um banco PostgreSQL via docker compose, com porta exposta para conexão local.

Cuidado com o docker.io
O pacote docker.io disponível no apt do Ubuntu é uma versão desatualizada e apresenta problemas no WSL2 (socket não responde, daemon não sobe). Use o script oficial da Docker.

1 Atualizar o sistema

Antes de instalar qualquer coisa, atualize os pacotes do sistema:

# Atualiza lista de pacotes
sudo apt update

2 Remover versão antiga (se instalada) Lição aprendida

Se você tentou instalar via apt install docker.io e teve problemas (daemon não subia, docker ps falhava), remova tudo antes de continuar: 

sudo apt remove docker.io docker-compose docker-compose-v2
Por que isso acontece?
O docker.io do apt não é mantido pela Docker Inc. No WSL2 ele tem dificuldade em iniciar o daemon (dockerd) porque o WSL não usa systemd por padrão. O script oficial da Docker lida com isso corretamente.

3 Instalar o Docker (método oficial)

Execute o script de instalação oficial da Docker:

curl -fsSL https://get.docker.com | sh

Isso instala a versão mais recente do Docker Engine, CLI e containerd automaticamente.

O que esse script faz
  • Adiciona o repositório oficial da Docker ao apt
  • Instala docker-ce, docker-ce-cli, containerd.io
  • Instala o plugin docker compose (V2 — sem hífen)

4 Adicionar seu usuário ao grupo docker

Por padrão, apenas o root pode usar o Docker. Para rodar sem sudo adicione seu usuário ao grupo docker: 

sudo usermod -aG docker $USER
Reinicie a sessão WSL
A alteração de grupo só tem efeito em novas sessões. Feche o terminal WSL completamente (ou rode newgrp docker) antes de testar.

Verifique que funcionou:

docker ps
# Deve retornar a lista vazia sem erro de permissão

5 Iniciar o daemon do Docker no WSL

O WSL2 não tem systemd ativado por padrão, então o Docker não sobe automático. Inicie o serviço manualmente: 

sudo service docker start

Para verificar se está rodando:

sudo service docker status
Dica: iniciar automático
Adicione sudo service docker start ao seu ~/.bashrc ou ~/.profile para não precisar rodar toda vez que abrir o WSL.

6 Criar o arquivo docker compose

No projeto, crie a pasta infra/ e dentro dela o arquivo compose.yaml com a configuração do PostgreSQL:

infra/compose.yaml
services:
  database:
    image: "postgres:16.0-alpine3.18"
    env_file:
      - ../.env.development
    ports:
      - "5432:5432"
.env.development
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
POSTGRES_USER=local_user
POSTGRES_DB=local_db
POSTGRES_PASSWORD=local_password
  • postgres:16.0-alpine3.18 — imagem leve baseada em Alpine Linux
  • POSTGRES_USER — define o nome do superusuário criado na inicialização (não é mais postgres)
  • POSTGRES_DB — banco criado automaticamente na inicialização
  • POSTGRES_PASSWORD — senha desse usuário
  • 5432:5432 — expõe a porta do container na máquina host
POSTGRES_USER muda tudo
Quando você define POSTGRES_USER, o PostgreSQL cria esse nome como superusuário na primeira inicialização — o role postgres padrão não é criado. Se tentar conectar com -U postgres, vai receber FATAL: role "postgres" does not exist.

7 Subir o banco de dados

Na raiz do projeto, execute:

# Sobe em background (-d = detached)
docker compose -f infra/compose.yaml up -d

Verifique que o container está rodando:

docker ps
# Deve exibir o container "database" com status "Up"

Para parar quando não precisar mais:

docker compose -f infra/compose.yaml down

8 Conectar ao PostgreSQL Pegadinha comum

Não use -U postgres
O usuário postgres só existe quando você não define POSTGRES_USER no env. Como o projeto define POSTGRES_USER=local_user, o superusuário criado é local_user. Conectar com -U postgres vai gerar: FATAL: role "postgres" does not exist.

Use sempre o usuário e banco definidos no .env.development: 

# Via Docker (sem precisar instalar psql no WSL)
docker compose -f infra/compose.yaml exec database psql -U local_user -d local_db

Ou, se tiver o psql instalado localmente:

psql -h localhost -U local_user -d local_db -p 5432
# Senha: local_password
Regra prática
O valor de -U deve sempre bater com POSTGRES_USER do seu .env. Se mudar o env, mude o comando de conexão também.

Dentro do psql, alguns comandos úteis:

\l          -- lista bancos de dados
\c nome_db  -- conecta a um banco
\dt         -- lista tabelas
\q          -- sai

Resumo rápido (TL;DR)

Sequência correta do zero
# 1. Remover versão antiga (se necessário)
sudo apt remove docker.io docker-compose docker-compose-v2

# 2. Instalar Docker oficial
curl -fsSL https://get.docker.com | sh

# 3. Adicionar usuário ao grupo docker
sudo usermod -aG docker $USER

# 4. Reiniciar sessão WSL, depois iniciar daemon
sudo service docker start

# 5. Subir o banco
docker compose -f infra/compose.yaml up -d

# 6. Conectar (use o POSTGRES_USER do seu .env, não "postgres")
docker compose -f infra/compose.yaml exec database psql -U local_user -d local_db

Troubleshooting

Problema

Cannot connect to the Docker daemon at unix:///var/run/docker.sock

Solução: o daemon não está rodando. Execute:

sudo service docker start

Problema

permission denied while trying to connect to the Docker daemon socket

Solução: seu usuário não está no grupo docker ou a sessão não foi renovada. 

# Verifique os grupos do usuário atual
groups

# Se "docker" não aparecer, rode e abra nova sessão
sudo usermod -aG docker $USER
newgrp docker

Problema

FATAL: role "postgres" does not exist

Solução: você está usando -U postgres, mas o projeto define um usuário diferente em POSTGRES_USER. Use o valor correto do seu .env.development: 

docker compose -f infra/compose.yaml exec database psql -U local_user -d local_db

O role postgres só é criado automaticamente quando POSTGRES_USER não está definido no env.

Problema

Porta 5432 já em uso

Solução: verifique o que está usando a porta:

sudo lsof -i :5432
