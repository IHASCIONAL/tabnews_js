Mailcatcher no desenvolvimento

Guia completo para o futuro eu — capturar e-mails sem enviar nada de verdade.

Contexto

Mailcatcher é um servidor SMTP falso que captura todos os e-mails enviados pela aplicação e os exibe numa interface web. Nenhum e-mail sai para a internet — tudo fica preso localmente. É a forma mais simples de testar fluxos de e-mail (confirmação de conta, recuperação de senha, notificações) sem correr o risco de enviar mensagens reais a usuários.

Porta Protocolo Para quê
1025 SMTP A aplicação envia e-mails aqui (em vez do 25 real)
1080 HTTP Interface web para visualizar os e-mails capturados
Por que não usar o Gmail/SMTP real no dev?
Credenciais reais em .env local são um risco, a taxa de envio do provedor pode bloquear testes em volume, e e-mails de teste podem chegar a usuários reais por engano. O Mailcatcher elimina todos esses problemas.

1 Instalar o Mailcatcher via RubyGems

Mailcatcher é uma gem Ruby. Instale-a globalmente no sistema com sudo para que o executável fique disponível em qualquer terminal: 

# Atualiza a lista de pacotes e instala o Ruby (se ainda não tiver)
sudo apt update
sudo apt install telnet

# Instala a gem mailcatcher globalmente
sudo gem install mailcatcher

2 Configurar no compose.yaml Método do projeto

No projeto usamos a imagem Docker sj26/mailcatcher, sem precisar instalar a gem manualmente. Adicione o serviço ao infra/compose.yaml:

infra/compose.yaml
services:
  database:
    container_name: "postgres-dev"
    image: "postgres:16.0-alpine3.18"
    env_file:
      - ../.env.development
    ports:
      - "5432:5432"

  mailcatcher:
    container_name: "mailcatcher-dev"
    image: "sj26/mailcatcher"
    ports:
      - "1025:1025"   # SMTP — a aplicação manda e-mail aqui
      - "1080:1080"   # Web UI — você lê os e-mails aqui

Suba (ou reinicie) os serviços para o container do Mailcatcher ser criado: 

docker compose -f infra/compose.yaml up -d

# Verifique que os dois containers estão rodando
docker ps
# Deve listar: postgres-dev  e  mailcatcher-dev
Nenhuma variável de ambiente necessária
A imagem sj26/mailcatcher não precisa de env vars — ela sobe imediatamente aceitando qualquer conexão SMTP na porta 1025.

3 Apontar a aplicação para o Mailcatcher

No .env.development, configure o host e a porta SMTP para apontar ao container local:

.env.development (trecho relevante) 
EMAIL_SMTP_HOST=localhost
EMAIL_SMTP_PORT=1025

Qualquer e-mail que a aplicação enviar via SMTP para localhost:1025 será capturado pelo Mailcatcher e nunca chegará a uma caixa de entrada real.

4 Acessar a interface web

Com o container rodando, abra o navegador em:

http://localhost:1080

Todos os e-mails capturados aparecem listados à esquerda. Ao clicar em um, você vê o conteúdo completo — assunto, destinatário, corpo em texto e HTML. A lista é limpa ao reiniciar o container.

Fluxo de teste
Dispare uma ação na aplicação que gere e-mail (cadastro, recuperação de senha…), depois abra localhost:1080 — o e-mail aparece ali em segundos.

5 Testar via Telnet Debug avançado

Você pode enviar um e-mail diretamente pelo protocolo SMTP usando telnet, sem depender da aplicação. Útil para confirmar que o servidor está escutando e para entender como o protocolo funciona. 

telnet localhost 1025
# Conexão estabelecida — o servidor responde:
220 EventMachine SMTP Server

# Apresente-se ao servidor
HELO ismael
250 Ok EventMachine SMTP Server

# Defina o remetente
MAIL FROM:<ismael@gmail.com>
250 Ok

# Defina o destinatário
RCPT TO: <ismaelhenriqueeu@gmail.com>
250 Ok

# Inicie o corpo do e-mail
DATA
354 Send it

# Escreva os headers e o corpo (linha em branco separa headers do corpo)
Subject: Teste por Telnet

Corpo do email.
# Um ponto sozinho na linha encerra o DATA
.
250 Message accepted

# Encerre a conexão
QUIT
221 Ok
Anatomia do protocolo SMTP
  • HELO — handshake inicial, identifica o cliente
  • MAIL FROM — envelope do remetente (não é o header From:)
  • RCPT TO — envelope do destinatário (pode repetir para múltiplos)
  • DATA — inicia a mensagem; encerra com . em linha sozinha
  • QUIT — fecha a conexão graciosamente

Após enviar, abra http://localhost:1080 — o e-mail de teste aparecerá imediatamente na interface.

Comandos úteis do dia a dia

Docker (método do projeto)

# Subir todos os serviços (banco + mailcatcher)
docker compose -f infra/compose.yaml up -d

# Ver logs do mailcatcher em tempo real
docker logs -f mailcatcher-dev

# Parar todos os serviços
docker compose -f infra/compose.yaml down

# Reiniciar só o mailcatcher (limpa os e-mails capturados)
docker restart mailcatcher-dev

Gem instalada localmente (instalação via sudo)

# Subir o servidor em background
mailcatcher

# Subir especificando as portas (padrão já é 1025/1080)
mailcatcher --smtp-port 1025 --http-port 1080

# Parar o servidor
pkill -f mailcatcher
# ou acesse http://localhost:1080/quit

Troubleshooting

Problema

telnet: Unable to connect to remote host: Connection refused

Solução: o container não está rodando. Verifique e suba: 

docker ps | grep mailcatcher
docker compose -f infra/compose.yaml up -d

Problema

E-mails não aparecem na interface localhost:1080

Solução: confirme que a aplicação está apontando para localhost:1025 no env. Verifique também os logs do container: 

docker logs mailcatcher-dev

Problema

Porta 1025 ou 1080 já em uso

Solução: descubra o processo que ocupa a porta:

sudo lsof -i :1025
sudo lsof -i :1080

Se for uma instância antiga do Mailcatcher rodando como gem local: 

pkill -f mailcatcher

Problema

sudo gem install mailcatcher falha por falta do Ruby

Solução:

sudo apt update
sudo apt install ruby ruby-dev build-essential
sudo gem install mailcatcher
